## 二、WIPO API 认证流程

```
1. GET https://branddb.wipo.int/api/search
   → 获取反爬页面，提取 uuid

2. GET https://api.branddb.wipo.int/captcha
   → 获取 ALTCHA 挑战 (SHA-256 工作量证明)
   → 403/429: 直接返回 False（不等待，云端IP等了也没用）
   → 200: 继续

3. altcha.solve_challenge_v1() 解题
   → 0.01-0.18 秒内完成

4. GET https://api.branddb.wipo.int/dbinfo?token=<payload>
   → 验证 token，建立会话

5. 设置 cookies: session_id = uuid
   → 后续 search 请求携带 HashSearch header
```

### 认证失败处理

- **captcha 403**：直接返回 `False`，不再重试等待。云端IP被WIPO风控时，等待15-75秒也无济于事。
- **dbinfo 失败**：返回 `False`
- 本地IP几乎不触发 captcha 403，所以本地技能脚本可靠性远高于云端

## 三、并发查询架构

### search() 方法流程

```
输入: term, offices=[US, CA, DE, FR, GB]

┌─ Attempt 0 (首轮) ──────────────────────────────────────────┐
│  ThreadPoolExecutor(max_workers=5)                          │
│  ┌─────────┬─────────┬─────────┬─────────┬─────────┐        │
│  │ US 查询 │ CA 查询 │ DE 查询 │ FR 查询 │ GB 查询 │        │
│  └────┬────┴────┬────┴────┬────┴────┬────┴────┬────┘        │
│       │         │         │         │         │              │
│  收集结果: 401/403 → retry_offices, 其他错误 → 警告, 200 → docs │
└──────────────────────────────────────────────────────────────┘

┌─ Attempt 1 (重试轮, 仅当有 retry_offices) ───────────────────┐
│  等待 2 秒                                                  │
│  重新 authenticate()                                        │
│  对 retry_offices 再并发查询                                 │
│  仍有失败的 → 加入 failed_offices                            │
└──────────────────────────────────────────────────────────────┘

返回: {response: {docs: [...]}, failed_offices: [...]}
```

### _query_office() 线程安全设计

每个办公室查询使用 `requests.post()` 直接调用（不通过 session），因为 `requests.Session` 不是线程安全的。手动复制 headers 和 cookies：

```python
headers = dict(self.session.headers)
headers["HashSearch"] = self.hash_search
cookies = self.session.cookies.copy()
r = requests.post(url, data=body, headers=headers, cookies=cookies, timeout=60)
```

返回值：
- `(office, data, None)` — 成功
- `(office, None, "401")` — 会话过期，需重认证
- `(office, None, "403")` — IP被风控，需重认证
- `(office, None, str(e))` — 其他错误

## 四、403 风控机制与应对

### 403 触发因素

| 因素 | 影响 |
|------|------|
| IP 类型 | 云端IP（Streamlit Cloud）高频触发；本地住宅IP几乎不触发 |
| 请求频率 | 间隔 < 1s 时风险增加 |
| 累积请求量 | 单次会话查询 > 100 词后概率上升 |
| captcha 端点 | 与 search 端点独立，captcha 403 阻塞认证流程 |

### 两个端点的 403 区别

```
captcha 端点 403:
  → 阻塞整个认证流程
  → authenticate() 返回 False
  → check_text() 抛出 RuntimeError
  → 无法继续查询，必须换IP

search 端点 403:
  → 只影响当前词的当前办公室
  → 记入 retry_offices
  → 2轮重试后仍失败 → 记入 failed_terms
  → 其他词继续查，不影响全局
```

### 应对策略

```
云端 (webapp):
  1. captcha 403 → 直接返回 False，不等待
  2. search 403 → 2轮重试（等2s重新认证），仍失败记入 failed_terms
  3. delay 默认 2s，自适应调节（成功多则降到0.5s）
  4. 重查1次（delay+1s），仍失败 → 显示失败词列表，引导转飞书助手

本地 (技能脚本):
  1. captcha 403 → 同样直接返回 False（但几乎不触发）
  2. search 403 → 同样2轮重试
  3. delay 默认 2s
  4. --terms 参数支持重查指定词条
```

## 五、failed_terms 追踪机制

### 数据结构

```python
# check_text() 返回值
{
    "risks": [
        {
            "term": "camping",
            "hits": 2,
            "examples": [
                {"brandName": "CAMPING", "office": "FR", "status": "Registered", ...},
                ...
            ]
        }
    ],
    "failed_terms": [
        {"term": "hello", "offices": ["DE", "FR"]},  # 这两个办公室403了
        {"term": "kitty", "offices": ["US", "CA", "DE", "FR", "GB"]},  # 全部失败
    ]
}
```

### 失败词来源

1. **search() 返回 failed_offices** — 2轮重试后仍有办公室未查到
2. **认证失败** — `client.authenticate()` 连续3次失败，该词所有办公室记入 failed_terms
3. **查询异常** — `client.search()` 抛出异常，该词所有办公室记入 failed_terms

### 重查逻辑（webapp）

```
首次查询完成
  ├── 有 failed_terms?
  │     ├── 是 → 显示「一键重查失败词」按钮
  │     │         ├── 点击 → check_text(terms=failed_terms, delay=delay+1)
  │     │         │          ├── 成功 → 合并结果
  │     │         │          └── 仍有失败 → 显示警告 + 可复制词条列表
  │     │         │                        「建议用飞书助手本地查询」
  │     │         └── retry_count = 1
  │     └── 否 → 正常显示结果
  └── retry_count >= 1?
        └── 是 → 不再显示重查按钮，直接显示失败词列表 + 飞书提示
```

关键设计：
- **重查只允许1次** — 云端IP被限流后重查多次也无意义
- **重查时 delay+1s** — 降低请求频率，提高成功率
- **传 terms 参数** — 绕过 extract_terms，避免短语被重新拆分
- **重查仍失败** → 显示可复制的词条列表，提示转飞书助手

## 七、词条提取逻辑

### extract_terms() 流程

```
输入文本
  ↓
1. strip_accents() — 归一化重音字符 (é→e, ü→u, ç→c)
  ↓
2. 提取单词 — regex [A-Za-z][A-Za-z0-9']*，≥3字母，过滤停用词
  ↓
3. 提取词组 — 按标点分段，每段生成 2-4 词 n-gram
   "Kool Fun" → ["kool fun"]
   "you hello kitty" → ["you hello", "hello kitty", "you hello kitty"]
  ↓
4. 去重 — 保持顺序，seen set 去重
  ↓
输出: ["word1", "word2", "phrase one", ...]
```

### 重查时传 terms 的意义

首次查询时，`extract_terms("hello kitty toy")` 会提取出 `hello`, `kitty`, `toy`, `hello kitty`, `kitty toy`, `hello kitty toy`。

如果 `hello kitty` 查询失败，重查时如果重新走 `extract_terms`，短语会被重新拆分。传 `terms=["hello kitty"]` 可以直接查这个短语，避免重复提取。

## 九、飞书助手集成

### 触发方式

用户在飞书中找到「商标风险排查助手」（WorkBuddy 应用），发送文本或词条，WorkBuddy 自动加载 `wipo-brand-checker` 技能执行查询。

### 本地查询优势

- 住宅IP几乎不触发 WIPO 403 风控
- 不受 Streamlit Cloud 资源限制
- 可直接创建飞书文档（--feishu）
- 适合查少量失败词（Stage 2）
